## Hypothesis

**PETSc `reason=-3` = `SNES_DIVERGED_LINEAR_SOLVE`.** The MUMPS LU linear solver
inside the Newton loop failed to produce a valid step at step 2 of the coarsest dt
level. The SNES residual trace confirms it: norm barely moves from
`3.221765760899e-06` to `3.221608448105e-06` in one Newton iteration (a ~0.005%
change), consistent with MUMPS returning a near-zero or garbage δu, after which
PETSc declares `SNES_DIVERGED_LINEAR_SOLVE`.

**Why step 2 of the coarsest level?**
The test uses `DT_LIST = [DT_BASE / 2**k for k in range(4)]` where
`DT_BASE = T_END / 16 = 1e-9 / 16 = 6.25e-11 s`.  The failure at
`t = 1.250e-10 s = 2 × 6.25e-11 s` exactly matches step 2 of `DT_LIST[0]`
(the coarsest level, processed first in the loop).  All four finer-dt runs
(DT_LIST[1–3]) complete successfully — the failure is specific to the largest
time step.

**Why is the Jacobian ill-conditioned at this step?**
The test docstring explicitly records the stability boundary: *"the dt window where
MUMPS stays stable on the post-ramp V_F/2 → V_F BC step (~50 ps minimum)"*.
`DT_BASE = 62.5 ps` sits only ~25% above that boundary.  The `bc_ramp_steps=10`
comment in the test config notes that the device was designed so the BC impulse
is small enough to keep MUMPS well-conditioned *"at DT_BASE = 6.25e-11 s"*.  The
failure suggests that estimate is no longer valid under the runner as it exists on
`main` after PR #50 — either the Slotboom Jacobian assembly changed in a way
that shifts the conditioning boundary upward past 62.5 ps, or the step-2 initial
condition (first BDF history point) is producing a near-singular mass-matrix
contribution at this dt scale.

**This regression was introduced by PR #50**, which rewrote
`tests/mms/test_transient_convergence.py` (new device: 1e17 doping, V_F=0.05 V,
new DT_LIST).  The CI run on that PR's merge commit (`1414c67`) also failed with
the same error; PR #50 was merged despite a red `docker-fem` job.

## What this PR does NOT do

This blocker document is the only change added here.  No speculative fix is
attempted:

- No changes to `semi/runners/transient.py` (Jacobian assembly, SNES tolerances,
  BC ramp logic).
- No changes to `tests/mms/test_transient_convergence.py` (DT_LIST, device
  parameters, ramp steps).
- No `@pytest.mark.xfail` added.

A fix will require a separate PR with an explicit scope:
either increasing `bc_ramp_steps` / reducing `DT_BASE` to move safely away from
the MUMPS conditioning boundary, or investigating whether the Slotboom Jacobian
at the coarsest dt level has a genuine defect introduced in PR #50.
