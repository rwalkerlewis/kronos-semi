# prompts/

This directory holds prompts generated by the **reviewer** agent for the
**worker** agent. See `.claude/agents/` for the agent definitions and
`CLAUDE.md` at repo root for the overall workflow.

## Conventions

- One markdown file per task. Name it after the task, not the date.
  Example: `m10_endpoints.md`, not `2026-04-23_prompt.md`.
- Keep prompts in the repo. They're review evidence — future
  contributors and AI agents can see what instructions a change was
  produced from.
- Purge periodically. Once a milestone ships and the PR merges,
  archive or delete the prompts that drove it. `git rm prompts/m9_*.md`
  after M9 is stable is fine; the git history preserves them.

## Anatomy of a good prompt

1. **Context** — branch, commit SHA, prior state.
2. **Task** — numbered steps, mechanical enough that Sonnet executes
   without judgment.
3. **Acceptance criteria** — exact commands whose output proves
   success.
4. **Constraints** — what NOT to touch.
5. **Reporting format** — what the worker reports back.

See `docs/IMPROVEMENT_GUIDE.md` §M9 and §M10 for examples of
well-formed milestone prompts.

## Why prompts are committed

- Auditable. A commit can reference the prompt file, making it clear
  what instructions produced the change.
- Reviewable. The reviewer agent re-reads its own prompts when
  auditing worker output.
- Reusable. Similar tasks can start from a past prompt as template.
